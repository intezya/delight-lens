## Цель

Переосмыслить продукт на основе фидбека Yasya: AI должен помогать **расследовать** проблемы, а не предлагать готовые решения. Сделать гипотезы менее категоричными, добавить альтернативные версии причин, разделить «подтверждённость проблемы» и «уверенность в решении», и пересобрать дашборд вокруг текущих проблем.

## P0 — Переосмысление гипотез

### 1. Модель данных (`src/lib/mock/data.ts`)
- Заменить `Insight` (одна гипотеза = одно решение) на структуру **Проблема → набор гипотез-причин**:
  - `Problem`: id, title, topicId, subtopicId, `problemConfidence` (подтверждённость проблемы), evidenceReviews, status, growth.
  - `CauseHypothesis` (≥2–4 на проблему): id, problemId, statement (возможная причина), `solutionConfidence` (уверенность в причине), `whatToCheck[]` (чек-лист расследования), `missingData[]`, `nextActions[]` (исследовательские, не «внедрить»), supportingReviews, alternativeRank.
- Добавить мок 3–4 проблем с 2–4 альтернативными гипотезами каждая (витринный товар → ошибка маркировки / продавцы не предупреждают / нет статуса в CRM / повреждённая упаковка).
- Убрать поля: `implementationTracking` с главной модели гипотезы (перенести в отдельную вкладку «Эффект»).
- Смягчить формулировки в моках: «возможно», «может быть связано», «требует проверки».

### 2. Страница гипотезы (`src/routes/insights.$insightId.tsx`)
- Вверху — **карточка проблемы** с `problemConfidence` («Проблема подтверждена: высокая, 76 отзывов»).
- Ниже — **переключатель альтернативных гипотез** (табы или вертикальный список с радио-картами): «Гипотеза 1 из 4».
- Для активной гипотезы — секции:
  1. Возможная причина (мягкая формулировка).
  2. **Что проверить перед внедрением** — чек-лист (новый компонент `WhatToCheck.tsx`).
  3. **Каких данных не хватает** — расширить существующий `NeededDataPanel`.
  4. Доказательства (existing `EvidenceList`).
  5. Уверенность в причине (отдельно от подтверждённости проблемы).
  6. Следующие исследовательские шаги (existing `NextSteps`, переписать формулировки).
- Sticky-панель действий (`StickyActions`) — заменить кнопки:
  - Было: «Принять / Передать в работу / Отклонить».
  - Станет: «Начать проверку», «Сформировать план исследования», «Запросить данные», «Связаться с клиентами», «Отклонить как нерелевантную».
- Кнопка «Сформировать план исследования» открывает Dialog с моковым 5-шаговым планом.

### 3. Карточки на страницах списков
- `src/components/insight-card.tsx`: показывать **проблему + количество альтернативных гипотез** («4 возможные причины»), `problemConfidence` отдельной плашкой.
- `src/routes/topics.$topicId.tsx`: группировка по подтемам остаётся, но внутри — карточки проблем, не отдельных гипотез.

### 4. Сценарий «Связаться с клиентом»
- Новый компонент `src/components/insight/CustomerFollowUp.tsx`: чек-лист (уточнить заказ / дату / магазин / запросить фото / зафиксировать результат) + кнопка «Открыть шаблон сообщения» (Dialog с моковым текстом).
- Доступен из StickyActions и из карточки отзыва (`review-drawer.tsx`).
- Явное предупреждение: «Это не публичный ответ, а follow-up для расследования».

## P1 — Дашборд и графики

### 5. Главный дашборд (`src/routes/index.tsx`)
- Переименовать «Утренний бриф за 30 дней» → **«Утренний бриф · последние 24 часа»** с переключателем периода (24ч / 7д / 30д), заголовок меняется по выбору.
- Оставить блоки: темы с ростом негатива, новые кластеры, критичные проблемы, гипотезы требующие проверки.
- Убрать с главной: эффект внедрений, прогресс воронки, delta sentiment — перенести во вкладку «Эффект» (`/impact`).

### 6. Темы (`src/routes/topics.tsx`)
- Добавить dropdown «Сортировать по: росту негатива / количеству отзывов / критичности / свежести».
- Разделить на две секции: **«Проблемные темы»** и **«Позитивные паттерны»** (вынести «Положительный опыт» в отдельный блок ниже).

### 7. Графики (`src/components/insight/ImpactTracker.tsx` и др.)
- Добавить заголовок, подписи осей, единицы измерения, легенду.
- Объяснить маркеры: круглая точка = внедрение, вертикальная линия = релиз, красная точка = всплеск негатива.
- Tooltip при hover: дата + событие + эффект.
- Убрать графики, которые не отвечают на конкретный вопрос.

### 8. Очистка тегов и моков
- Унифицировать badges на карточках: только `{N} отзывов`, `+X% за неделю`, `{N} источников`, `Новый кластер`.
- Удалить «сигнал 88», непонятные числовые теги.
- Пройтись по `settings.tsx` — удалить нефункциональные тогглы (сарказм и т.п.) или объяснить через `InfoHint` что они меняют.

## Что НЕ делаем в этой итерации

- Автоответы на отзывы (намеренно, по фидбеку Yasya).
- Реальные интеграции (всё мок).
- Внешний контекст (релизы, сезонность) — P2/Could-have.
- «Позитивные паттерны для масштабирования» как отдельная фича — только разделение в списке тем.
